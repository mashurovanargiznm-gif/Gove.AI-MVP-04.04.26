### DB SCHEMA (Assume Supabase table `transactions` has these columns)
id (uuid), created_at (timestamp), debtor_bin (text), amount_kzt (numeric), description (text), knp_code (text), oked_code (text), receiver_iin (text), ai_decision (text), withheld_percent (integer), ai_reason (text), solana_signature (text), status (text: 'pending', 'approved', 'blocked').

### APP ARCHITECTURE & LOGIC
1. API `/analyze` (Incoming Payment Simulation):
   - Receives: amount, description, knp_code, oked_code, receiver_iin.
   - Logic: Prompt Gemini 1.5 Flash. 
     * Rule 1 (White): KNP for "Salary" or "Taxes" -> 0% split (allow 100%).
     * Rule 2 (B2B Normal): Valid B2B payment -> 15% split.
     * Rule 3 (Fraud): Mismatch between KNP and OKED (e.g., KNP "Consulting" but OKED "Construction") OR receiver_iin == "010101010101" -> 100% Block.
   - Gemini returns strict JSON: {"decision": "allow/split/block", "percent": int, "reason": "Explainability text in Russian"}.
   - Save to Supabase with status 'pending'. Return success to UI.

2. API `/approve` (ChSI Action):
   - Receives: transaction_id.
   - Logic: 
     - Calculate withheld amount in micro-SOL based on `percent`.
     - Execute real Solana Devnet transfer using `solders` from SOLANA_PRIVATE_KEY to CREDITOR_PUBKEY.
     - Send Telegram message: "⚖️ Gove.AI | ЧСИ Утвердил. Удержано: {percent}%. БИН: {debtor_bin}. TX: {solana_signature}".
     - Update Supabase status to 'approved' or 'blocked'.

3. API `/transactions` (Live Feed):
   - Fetch all transactions from Supabase ordered by created_at DESC.
